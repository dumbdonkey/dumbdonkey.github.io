使用keep-alive特性让连接保持空闲，然后比对BIO与NIO的差异

各种Connector之间的差异

BIO

传统阻塞IO

All I/O operations are blocking in processor thread
● SSL handshake
● Read request line (e.g. GET, POST, etc.)
● Read request body
● Write response
● Read next request (HTTP keep-alive)
● Simple, stable, mature

请求量受制于线程数.

NIO

Single thread handles request after request-line
● Poller thread(s) manage non-blocking Selector
● Read SSL handshake
● Read request line
● Wait for next keep-alive request

源码分析

  • 整体流程

  • 启动部分

    1. 启动container

    2. 使用ServerSocketChannel绑定端口进行监听

    3. 启动acceptor线程，acceptor线程由各EndPoint进行创建 (NioEndPoint,BIOEndPoint,Nio2EndPoint)

    4. 根据配置的数目创建accept线程同时去监听服务端口 (serversocket默认为 blocking模式）


请求处理

请求到达

有新的连接到达时,通过setSocketOptions方法将连接交给一个poller进行处理

创建一个NioSocketWrapper
构建一个PollerEvent对象,事件为（OP_REGISTER）加入events队列
(此处PollerEvent对象时复用的，每次使用完毕后调用reset方法重置，然后重新放入events队列).

  • PollerThread: 完成accept线程和工作线程间的dispatch工作

    每个poller都有自己的Selector实例,(负责监听读写事件，当这些事件发生时，交给工作线程做真正的处理)

    从events队列pull事件。

    对于OP_REGISTER事件将会注册OP_READ事件到Selector上

    然后poller线程执行select操作，等待事件触发

    不断pollEvent，逐一处理

Http请求处理阶段(SocketProcessorBase)

响应输出部分

可以看到NIOEndpoint是blocking write

NIO2

Single thread handles request after request-line
● The thread are handled via an
AsynchronousChannelGroup and completion call backs
● Read SSL handshake
● Read request line and headers
● Wait for next keep-alive request

● NIO2 implementation takes care of blocking using Future
objects waiting for IO
● Request body reads
● Response writes
● Processor thread sleeps during blocking
● Uses JSSE for SSL/TLS
● It emulates sendFile (NIO1 transferTo doesn’t work with
NIO2)
